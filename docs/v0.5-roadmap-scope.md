# TERRY-LIS v0.5 路线图与范围说明

## 1. 基线状态

v0.4.0 RC 1 已作为 GitHub Pre-release 发布。Release assets 已包含 installer ZIP 和 portable ZIP。

最终回归已通过十个 smoke 脚本，以及 `db:reset` / `db:check`。当前基线已经覆盖样本接收、结果审核、危急值闭环、报告输出与发布、系统规则、质控事件、试剂近效期预警和传染病阳性预警处理等核心写入闭环。

该应用当前仍是桌面端原型系统，不是生产级医疗 LIS。所有功能验证均应理解为原型阶段的流程、数据写入、审计留痕和桌面端可运行性验证。

## 2. v0.5 主题

v0.5 的主题是从“流程能跑”进入“系统行为受治理”。

本阶段重点关注权限边界、操作治理、报告交付增强、AI 插件化准备。目标不是扩大业务范围，而是在 v0.4 RC 已经跑通的基础上，让关键操作更可控、更清晰、更容易验证。

## 3. v0.5 目标

- 为敏感操作建立角色与权限边界。
- 明确样本签收、结果审核、危急值通知、报告发布、质控处理、试剂近效期处理、传染病阳性预警处理、系统规则变更等动作的授权规则。
- 增强报告交付、导出、打印相关流程的安全性与清晰度。
- 为 AI 预审能力插件化做准备，但暂不依赖真实 AI 服务商。
- 增强审计可见性和操作员侧约束。
- 保持 v0.4 RC 稳定基线，不破坏既有 smoke 验证。

## 4. 非目标

- 不用于真实患者生产环境。
- 不接入真实医院 HIS / LIS / EMR。
- 不接入真实检验仪器。
- 不做自动诊断。
- 不实现真实电子签名 / CA 签章。
- 不做支付、云同步、多租户 SaaS。
- 不做自动更新系统。
- 不声称具备生产级网络安全或医疗合规认证。

## 5. v0.5 建议工作包

1. v0.5.1 权限模型与敏感操作矩阵
2. v0.5.2 敏感操作的 UI gating
3. v0.5.3 审计日志可见性与操作追踪增强
4. v0.5.4 报告导出 / 打印流程增强
5. v0.5.5 AI 预审插件接口准备
6. v0.5.6 v0.5 最终回归与 RC checklist

## 6. 建议 PR 顺序

每个工作包都应该通过小型 GitHub PR 交付。一个 PR 只解决一个关注点，避免把权限、UI、数据库、报告和 AI 插件准备混在同一个变更里。

没有明确计划，不做 schema 变更。不提交 `dist` 构建产物。不提交 `data` 运行时数据库文件。每个行为变更都必须有 smoke 或人工验证路径。

## 7. v0.5 验收标准

- v0.4 既有 smoke 测试继续通过。
- 新增权限 / 操作治理行为有文档说明并经过验证。
- 敏感操作能在角色或操作员上下文中被明确约束。
- 报告导出 / 打印流程更清晰、更安全。
- AI 插件化边界被文档化；代码实现只在后续实现 PR 中最小化引入。
- 发布 v0.5 RC 前准备 release notes 和 checklist。

## 8. 风险控制

- 保留 v0.4.0 RC 1 作为回滚基线。
- 避免大型混合 PR。
- schema 和 seed 变更必须少而明确。
- 不提交 `dist`、运行时数据库、临时日志、下载的 release assets。
- 所有医疗内容都仅视为 demo 数据。
