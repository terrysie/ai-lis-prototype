# TERRY-LIS v1.0 Artifact And GitHub Release Plan

This plan defines how the final v1.0 assets are controlled after a release
candidate passes the required checks. It does not create an archive, a tag, or a
GitHub Release.

## Current Build Configuration Facts

The current `package.json` uses the Windows build command:

```powershell
npm.cmd run build
```

It invokes `electron-builder --win` with the `nsis` Windows target. The output
directory is `dist`, and the configured artifact template is:

```text
TERRY-LIS-${version}-Windows-${arch}.${ext}
```

Therefore, the expected installer naming rule for a successful current build is:

```text
TERRY-LIS-<package-version>-Windows-<arch>.exe
```

The configuration has no independent portable target. A portable ZIP is a
controlled derived asset made from the successful `dist/win-unpacked` runtime;
it is not an electron-builder portable build target.

## Final Asset Rules

| Asset | Required naming rule | Source | Release condition |
| --- | --- | --- | --- |
| Windows installer | `TERRY-LIS-<package-version>-Windows-<arch>.exe` | Current NSIS build output in `dist/` | Only after build and installer tests pass. |
| Portable ZIP | `TERRY-LIS-<package-version>-Windows-<arch>-portable.zip` | Contents of the successful `dist/win-unpacked` runtime | Only after portable-startup test passes. |

The archive root must contain the unpacked runtime contents, including
`TERRY-LIS.exe` and its required runtime directories. Do not wrap those contents
in an additional `win-unpacked` parent folder.

Do not include source files, a nested `dist/` folder, `data/`, SQLite files,
build logs, historical artifacts, test exports, unreviewed ZIP files, or other
unrelated files in release assets.

## Portable ZIP Procedure

After the candidate Windows build succeeds and `dist/win-unpacked` has passed
the unpacked-runtime inspection, set the actual version and architecture in the
following controlled procedure:

```powershell
$unpacked = "dist\\win-unpacked"
$portable = "dist\\TERRY-LIS-<version>-Windows-<arch>-portable.zip"
Compress-Archive -Path "$unpacked\\*" -DestinationPath $portable -Force
```

Before release, inspect the archive contents and run the extracted application
from a location outside the build directory. Do not create the archive until the
RC checklist has reached its artifact stage.

## SHA-256 Record And Verification

Generate a SHA-256 for each approved final asset, for example:

```powershell
Get-FileHash -Algorithm SHA256 -LiteralPath "dist\\TERRY-LIS-<version>-Windows-<arch>.exe"
Get-FileHash -Algorithm SHA256 -LiteralPath "dist\\TERRY-LIS-<version>-Windows-<arch>-portable.zip"
```

Record the emitted hash together with the exact asset name, candidate commit,
and build date in the RC evidence record. After GitHub upload, download every
asset again to a separate folder and run the same `Get-FileHash` command. The
downloaded value must exactly match the recorded value before Release approval.

## GitHub Release Asset Plan

The final Release should contain only the approved installer and portable ZIP,
plus release notes that name the corresponding SHA-256 values. Do not upload
build logs, source data, database files, test artifacts, or unreviewed files.

Prepare the Release title and tag only after the candidate, package version,
assets, hashes, and Go decision are confirmed:

| Field | Preparation rule |
| --- | --- |
| Title | `TERRY-LIS v<package-version>` |
| Tag | `v<package-version>` |
| Body | Start from `docs/v1.0-release-notes-draft.md`, replace all placeholders, and keep the local-Demo limitations. |
| Assets | The final installer and portable ZIP from the approved asset manifest. |

## Human Final Confirmations

The following actions require human confirmation and must not be performed as
part of this documentation preparation:

1. Approve the candidate commit, version, build output, and asset names.
2. Open the unpacked app, install and launch the NSIS installer, uninstall it,
   and start the extracted portable runtime on Windows.
3. Confirm the Demo Guide, walkthrough, core workflow, alerts, and permission
   flows against the final candidate.
4. Create or approve the final tag and GitHub Release, upload approved assets,
   then verify downloaded SHA-256 values.
