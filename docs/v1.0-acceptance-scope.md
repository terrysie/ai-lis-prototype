# TERRY-LIS v1.0 Acceptance Scope

## 1. v1.0 Product Positioning

TERRY-LIS v1.0 is a Windows local LIS workflow Demo / prototype. It must be installable or runnable as a portable package, use example data, and demonstrate the main laboratory workflow end to end.

v1.0 does not connect to real hospital systems or laboratory instruments, does not process real patient data, and must not be presented as a hospital production system, clinical decision system, or real laboratory reporting system.

## 2. Audit Snapshot and Evidence Rules

- Audit baseline: `origin/main` commit `569fd8c`, which includes PR #50 and the completed v0.6 closeout.
- `package.json` declares Electron `main.js`, version `1.0.0`, an NSIS Windows build target, local SQLite support, `db:reset`, `db:check`, and ten workflow smoke scripts.
- `index.html`, `main.js`, `preload.js`, `src/database/**`, the existing RC documents, and the v0.6 guides are implementation evidence.
- Historical RC notes and README statements are historical verification evidence only. They do not prove the current v1.0 release candidate still passes.
- A v1.0 acceptance item is complete only when its required evidence is current for the release candidate commit. Existing implementation without a current rerun is marked partial.

## 4. Items Outside v1.0

- Real hospital HIS, LIS, EMR, PACS, or instrument integrations.
- Real patient data, real clinical reports, clinical decision use, or hospital production deployment.
- Production identity management, production authorization, compliance certification, electronic signatures, or CA report signing.
- Multi-terminal production database design, cloud synchronization, auto-update, release signing, or printer/instrument integration.

## 5. Remaining Route

### PR 1: v1.0 Release-Control Documentation

Codex prepares a single release-control document set: v1.0 RC checklist, release-notes draft, final user run/reset guidance, portable ZIP assembly procedure, artifact naming decision, and SHA256 recording template. It must not change runtime logic, schema, seed, or packages.

### PR 2: v1.0 RC Remediation (Conditional)

Run the fresh verification gates on a candidate commit. Codex fixes only reproducible release-blocking defects found by smoke, build, or manual acceptance. Any required package, schema, seed, or architectural change must be separately approved before work begins.

### Final Stage: v1.0 RC Verification and Release

Run the complete regression, build Windows artifacts, perform manual installer and portable checks, calculate hashes, review release-control documents, merge approved PRs, and create the GitHub Release. This is a release-control stage, not a feature-development stage.

This plan uses at most two additional development/documentation PRs after the acceptance-scope PR, followed by one RC verification and release stage.

## 7. Release Gates

The v1.0 release candidate cannot be approved until all of the following are true:

1. Candidate `main` is synchronized and the worktree is clean; no `dist/`, `data/`, database, export, or build-log file is tracked.
2. `npm.cmd run db:reset` and `npm.cmd run db:check` pass on the candidate.
3. All ten smoke scripts pass on the candidate and their runtime data is reset afterward.
4. `npm.cmd run build` produces the expected NSIS installer and unpacked runtime without committing `dist/`.
5. A human verifies installer install/start/uninstall and the portable ZIP start path on Windows.
6. Installer and portable ZIP SHA256 values are recorded and checked against the uploaded GitHub Release assets.
7. README, user guidance, v1.0 RC checklist, release notes, and known limitations match the final artifact names and version decision.
8. GitHub Release text retains the local-Demo, example-data, and non-production boundary.

## 8. Current Completion Calculation

The matrix contains 18 acceptance items: 0 complete, 15 partial, and 3 not complete.

For planning, complete items count as 1, partial items as 0.5, and not-complete items as 0. The current formal v1.0 acceptance completion is therefore `(0 + 15 × 0.5) / 18 = 41.7%`, expressed externally as approximately 42%.

The largest gaps are current-candidate regression evidence, reproducible Windows installer/portable delivery with hashes, and final release-control artifacts. The current implementation and functional coverage are clearly broader than 42%; 42% measures current-candidate release acceptance evidence readiness, not the number of implemented code features. The score is deliberately based on release acceptance evidence rather than historical claims.
