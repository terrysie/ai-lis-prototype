# TERRY-LIS v1.0 Interface & Reliable Communication Core

## 定位与边界

本模块是 Windows 本地 LIS 能力证明，不是生产接口平台。所有端点仅监听 `127.0.0.1` 或本地目录，所有身份、申请、样本和结果均为 deterministic synthetic data。实现不连接真实医院、仪器或患者数据，不声明兼容任何厂商私有协议，也不构成医疗认证、合规结论或医院验收声明。

## 架构

数据流分为五层：

1. Electron `index.html` 的 Interface Center 提供操作与追踪界面。
2. `preload.js` 只暴露明确的 IPC 方法，renderer 不直接访问 Node.js 或数据库。
3. `main.js` 负责 IPC 路由，并启动只监听本机的 HTTP、TCP 和 file-drop runtime。
4. `src/interface/interfaceCore.js` 提供适配器注册、持久化队列、幂等、处理器、重试、dead-letter、模拟器和业务闭环。
5. `sql.js` 将每次业务状态、消息、尝试、连接、投递与审计记录导出到本地 SQLite 文件。

核心业务不依赖特定 adapter。adapter 通过协议、parser/formatter、连接配置、字段映射和 retry policy 注册；新增 adapter 不需要修改样本或结果审核流程。

## Adapter 模型

每个 adapter 持久化以下信息：稳定 `adapter_id`、名称、类型、protocol、direction、enabled、connection JSON、parser、formatter、health、最近通信时间、retry policy 及 capability label。`interface_mappings` 独立保存 external field、local field、transform/code map 和启用状态。

内置 adapter：

| Adapter | 类型 | 本地通道 | 标记 |
|---|---|---|---|
| `his-local` | HIS simulator | HL7 v2 / HTTP | simulator capability proof |
| `emr-local` | EMR/EHR simulator | internal mock | simulator capability proof |
| `exam-local` | 体检 simulator | file-drop | simulator capability proof |
| `billing-local` | 计费 simulator | REST / HTTP | simulator capability proof |
| `cbc-synthetic` | 血液分析仪 | ASTM-like / TCP | synthetic profile |
| `bio-synthetic` | 生化分析仪 | HL7 v2 / internal mock | synthetic profile |
| `immuno-synthetic` | 免疫分析仪 | file-drop | synthetic profile |

## 能力证明协议

- HL7 v2 必要子集：ORM/O01 类申请、ORU/R01 类结果、ACK，以及 PID/ORC/OBR/OBX 的患者、申请、样本、项目、结果、单位、参考范围、异常标志、结果时间和状态字段。
- ASTM-like：公开标准思路下的 STX/ETX/EOT framing、两位十六进制 checksum、H/P/O/R/L record parsing。该实现仅作能力证明。
- HTTP：`127.0.0.1:17771/health`、`POST /hl7` 和 `POST /billing`。
- TCP：`127.0.0.1:17772`，接收 ASTM-like frame 并返回 ACK/NAK 控制字符。
- file-drop：Electron userData 下 `data/interface-drop/exam` 和 `data/interface-drop/immuno`。输入 `.json` 被改名为 `.processed.json`，同时生成 `.ack.json` 或 `.error.json`。
- internal mock：用于无需真实中间件的 deterministic outbound ACK 和模拟器闭环。

## 可靠消息状态机

`received/queued → processing → succeeded` 是正常路径。处理失败后，根据 retry policy 进入 `retrying`，记录 `last_error` 和 `next_retry_at`；达到 `max_attempts` 后进入 `dead-letter`。人工可将 `retrying`、`failed` 或 `dead-letter` 消息重新排队处理，也可取消尚未成功的消息。

每个处理尝试独立写入 `interface_message_attempts`。消息保存 direction、trace id、source、destination、protocol、raw payload、normalized payload、关联 sample/order/report id、尝试次数和唯一 idempotency key。重复 idempotency key 返回原消息，不重复创建消息或业务数据。

当前 worker 是进程内、SQLite 持久化 worker。IPC、HTTP、TCP、file-drop 和一键演示均调用同一处理入口；它不是外部消息中间件。

## Simulator 与故障注入

HIS 可创建 synthetic 检验申请并接收 ACK；EMR/EHR 和体检端接收发布报告；计费端接收 synthetic 状态；三类仪器根据场景生成固定、可复现的正常、异常或危急值结果。故障模式包括 `success`、`timeout`、`error`、`duplicate`、`invalid-field` 和 `disconnect`。故障模式变更写入 `audit_logs`。

结果规则固定，例如生化 K 为 4.2/5.9/6.8，血液 WBC 为 7.1/18.4/35.2，免疫 HBsAg 为 0.2/1.4/18.6。相同 scenario key 生成相同 order/sample/idempotency key，便于证明重复发送不会重复落库。

## 完整演示路径

1. 打开“接口中心”，确认 7 个 adapter 及健康状态。
2. 在 Simulator Lab 选择仪器和结果类型，点击“一键执行完整闭环”。
3. HIS simulator 生成 HL7 ORM，消息中心持久化 inbound message。
4. processor 创建 `laboratory_orders`、order item 和现有 `samples` 记录。
5. LIS 创建 instrument outbound message；synthetic instrument 返回 deterministic ORU。
6. processor 将结果写入现有 `test_results`、`result_reviews`，危急值结果同时写入 `critical_values`。
7. 一键演示执行 synthetic 审核/发布留痕，创建 report outbound message。
8. EMR simulator 返回 ACK，`external_report_deliveries` 保存 trace、delivery status 和 ACK。
9. 界面自动用 trace id 筛选完整消息链，可查看 raw、normalized payload 和每次 attempt。
10. 使用同一个 scenario key 再执行时，订单、结果和投递不会重复创建。

普通结果审核页的“正式发布报告”也会创建 outbound report、接收本地 EMR ACK，并在结果详情显示 delivery status、ACK 和 trace id。

## 数据库与审计

新增表：`laboratory_orders`、`laboratory_order_items`、`interface_adapters`、`interface_mappings`、`interface_messages`、`interface_message_attempts`、`interface_connections`、`simulator_scenarios`、`external_report_deliveries`。

扩展 `samples`：`external_patient_id`、`external_order_no`、`external_status_code`、`interface_trace_id`。扩展 `test_results`：`interface_message_id`。

adapter 启停、mapping 修改、message retry/cancel、dead-letter reprocess、simulator fault injection 及 interface demo 的审核发布动作均写入现有 `audit_logs`。外部报告重发通过同一消息 retry/reprocess 路径留痕。

## 自动测试

- `npm run test:interface-protocols`：HL7 ORM/ORU/ACK、ASTM framing/checksum/parsing、field mapping。
- `npm run test:interface-core`：adapter registry、mapping 编辑与审计、消息幂等、指数退避、dead-letter、手工重处理、取消、HIS/仪器/报告集成。
- `npm run smoke:interface-e2e`：HIS → LIS → instrument → LIS → EMR 完整闭环、危急值规则、重复申请/结果幂等和 trace/ACK。

## 已知限制与真实对接仍需资料

当前没有 TLS、OAuth/mTLS、真实用户认证、网络分区协调、跨进程锁、集群 worker、生产调度、消息签名、PHI 脱敏策略或医院级监控告警。`sql.js` 文件导出模式适合本地能力证明，不适合多实例生产并发。

接入真实系统前仍需：厂商/医院批准的接口规范与版本、消息样例、字段字典与编码表、字符集、时区、ACK/错误语义、重试与超时 SLA、网络地址/端口/TLS 证书、身份认证材料、样本与项目主数据映射、异常/危急值规则、数据保留与审计政策、测试环境和验收用例。任何私有协议兼容性必须基于这些资料单独实现和验证。
