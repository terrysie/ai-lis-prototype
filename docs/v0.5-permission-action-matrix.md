# TERRY-LIS v0.5 权限模型与敏感操作矩阵

## 1. 设计目标

v0.5 的权限模型目标不是做完整生产级 IAM，而是为桌面端原型建立清晰的角色边界和敏感操作约束。

权限设计要服务于后续 UI gating、后端 action guard、audit log、人工验证和 smoke 测试。权限模型应让每个敏感动作的执行主体、确认要求、留痕要求和验证方式更明确。

所有医疗数据仍然是 demo 数据，系统不用于真实患者生产环境。

## 2. 角色模型

- `system_admin`
  系统管理员。负责系统设置、规则配置、用户/权限相关管理。不能默认替代检验审核职责。

- `lab_manager`
  实验室主管。负责关键审核、质控处置、异常流程监督。

- `technician`
  检验技术人员。负责样本签收、退样、补采任务、部分质控事件处理。

- `reviewer`
  结果审核人员。负责 AI 预审后的结果审核、报告确认。

- `nurse_or_clinician`
  临床或护理通知接收角色。主要用于危急值通知接收或确认，不应拥有检验结果修改权限。

- `viewer`
  只读查看角色。只能查看允许范围内的数据，不允许执行写入动作。

- `demo_operator`
  原型演示默认操作员。用于当前桌面 demo，但它不是生产权限模型，不应被理解为真实部署中的通用管理员或默认授权角色。

## 3. 敏感操作定义

以下动作属于 v0.5 需要纳入权限边界和操作治理的敏感操作：

- 样本签收确认
- 样本退样
- 创建补采任务
- 结果审核通过
- 结果审核退回
- 危急值通知记录
- 危急值闭环确认
- 报告预览
- 报告发布
- 报告导出
- 报告打印
- 质控事件处理
- 试剂近效期预警处理
- 传染病阳性预警处理
- 系统规则新增
- 系统规则编辑
- 系统规则启用 / 停用
- AI 预审配置变更

## 5. v0.5 实施边界

- 当前 PR 只新增设计文档，不实现代码。
- 后续实现 PR 可以先从 UI gating 做起，再进入 action guard。
- schema 是否变化必须单独评估，不在本步骤处理。
- 不引入真实登录系统。
- 不引入云端用户体系。
- 不引入生产级身份认证。

## 6. 后续实现建议

1. 第一阶段：前端按钮级提示与禁用状态。
2. 第二阶段：preload / main action 层增加简单 role guard。
3. 第三阶段：audit log 显示操作者、动作、结果、时间。
4. 第四阶段：为敏感动作补充 smoke 或人工验证路径。
5. 第五阶段：再考虑是否需要 schema / seed 扩展。

## 7. 验收标准

- 文档明确列出角色。
- 文档明确列出敏感操作。
- 文档给出完整权限矩阵。
- 文档明确 `demo_operator` 只是原型演示角色。
- 文档明确本步骤不做代码实现。
- 不修改业务代码、数据库、package、dist、data 或运行时文件。

## 8. 风险控制

- 不把 `demo_operator` 误认为生产权限角色。
- 不把 UI gating 误认为安全认证。
- 不在一个 PR 中混入权限、数据库、报告、AI 多类变更。
- 不承诺生产级医疗合规。
- 不处理真实患者数据。
