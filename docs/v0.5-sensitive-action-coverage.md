# TERRY-LIS v0.5 敏感操作覆盖清单

## 本步骤目标

本步骤为 TERRY-LIS v0.5 增加一个可见的敏感操作覆盖清单，用于说明当前 desktop demo 中哪些敏感动作已经具备 UI gating、main action guard、audit log 留痕预期和 demo_operator 边界提示。

本步骤不新增业务写入能力，不引入真实登录系统，不改变数据库结构，也不改变既有后端写入逻辑。

## 覆盖清单字段说明

- `actionKey`：前端 UI gating 和 main action guard 使用的动作标识。
- `moduleName`：动作所属业务模块。
- `actionName`：面向用户展示的敏感动作名称。
- `uiGatingStatus`：表示前端是否已有提示和二次确认入口。
- `actionGuardStatus`：表示 main 层是否已有最小入口 guard，或仅配置预留。
- `auditLogStatus`：表示该动作当前是否应写入 `audit_logs`。
- `demoOperatorStatus`：说明 `demo_operator` 在桌面 demo 中的边界。
- `backendStatus`：说明当前是否有真实后端写入、仅导出、UI 预留或未接入。
- `note`：简要说明当前处理方式和限制。

## 已覆盖动作列表

- `sampleReceptionConfirm`：样本签收确认
- `sampleReceptionReject`：样本退样
- `sampleRecollectionCreate`：创建补采任务
- `resultReviewApprove`：结果审核通过
- `resultReviewReject`：结果审核退回
- `criticalValueNotify`：危急值通知记录
- `criticalValueAcknowledge`：危急值临床确认
- `criticalValueComplete`：危急值闭环确认
- `reportPublish`：报告发布
- `reportExport`：报告导出
- `reportPrint`：报告打印
- `qcEventHandle`：质控事件处理
- `reagentExpiryHandle`：试剂近效期预警处理
- `infectiousAlertHandle`：传染病阳性预警处理
- `systemRuleCreate`：系统规则新增
- `systemRuleEdit`：系统规则编辑
- `systemRuleToggle`：系统规则启用 / 停用

## UI gating 与 action guard 的边界

UI gating 是前端层面的提示、二次确认和 demo_operator 边界说明，主要用于让演示用户在执行敏感动作前看到操作风险和审计留痕提示。

action guard 是 main 层的最小入口保护，用于在调用数据库写入函数前统一经过敏感动作入口。当前实现仍处于 desktop demo 模式，不等同于生产级认证、授权或审计合规系统。

`reportPrint` 当前没有真实后端打印动作，只作为敏感动作预留。`systemRuleCreate` 当前只在 UI 中展示原型提示，不新增 `system_rules` 写入。

## demo_operator 的限制

`demo_operator` 仅用于当前桌面端原型演示。它不是生产权限角色，也不应被理解为真实部署中的默认管理员、审核员或通用授权用户。

在真实系统中，样本接收、结果审核、危急值闭环、报告发布、质控处理、试剂预警处理、传染病阳性预警处理和系统规则变更都必须结合真实用户身份、角色权限、岗位职责和审计制度进行约束。

## 当前不是生产级认证系统

当前覆盖清单只用于 v0.5 原型阶段的操作治理可视化。它不提供真实登录、不提供生产级权限认证、不提供医院合规审计结论，也不适用于真实患者生产环境。

## 人工检查清单

1. 打开系统设置页面，确认可见“敏感操作覆盖清单”区块。
2. 确认 summary cards 显示总敏感动作、UI gating、action guard、audit log 和预留/无后端统计。
3. 确认表格能看到 UI gating、action guard、audit log、demo_operator 和后端状态。
4. 确认 `reportPrint` 标记为预留或未接入，不出现新增打印写入能力。
5. 确认 `systemRuleCreate` 标记为 UI 预留，不出现新增系统规则写入能力。
6. 确认中文显示正常，无乱码。
7. 确认原有导航、样本接收、结果审核、危急值、报告发布、系统规则、质控和预警处理按钮行为不变。
8. 确认既有 smoke 测试和 `db:reset` / `db:check` 仍通过。
