# TERRY-LIS v0.5 RC 候选前检查清单

## 1. 本文档目的

本文档用于 TERRY-LIS v0.5 RC 候选前的总回归检查。它是 release control 和人工验收辅助清单，不是 release notes，不是生产部署说明，也不代表当前系统已经达到真实医疗生产环境要求。

本清单用于确认 v0.5 已完成的权限治理、敏感操作提示、main 层最小 action guard、敏感操作覆盖清单和既有写入闭环在 RC 候选前仍然可验证、可追溯、可检查。

## 2. 当前 v0.5 已完成能力

- v0.5 roadmap scope：已形成 v0.5 路线图与范围说明，明确从“流程能跑”进入“系统行为受治理”。
- 权限模型与敏感操作矩阵：已文档化角色模型、敏感动作、建议允许角色、audit log 要求、二次确认要求和 demo_operator 边界。
- UI gating：`index.html` 已加入 `sensitiveActionGating` 和 `confirmSensitiveAction`，为敏感动作提供前端提示与确认。
- main action guard：`main.js` 已加入 `defaultActionGuardContext`、`sensitiveActionGuards` 和 `guardSensitiveAction`，在 main 层为现有敏感写入入口提供最小 guard。
- 敏感操作覆盖清单：系统设置页已展示“敏感操作覆盖清单”，用于可视化 UI gating、action guard、audit log、demo_operator 和后端状态。
- demo_operator 边界提示：首页和敏感操作确认提示中已明确 `demo_operator` 仅用于桌面 demo，不是生产权限角色。
- audit log 预期说明：覆盖清单和权限矩阵已说明哪些现有写入动作应写入 `audit_logs`，以及哪些动作只是 UI 预留或当前无真实写入。

## 3. 自动验证清单

RC 候选前必须人工执行以下命令，并确认全部通过或输出符合预期：

```powershell
git status -sb
git branch --show-current
git log --oneline --decorate -8
git ls-files --others --exclude-standard
git diff -- data/
npm.cmd run db:reset
npm.cmd run db:check
npm.cmd run smoke:sample-reception
npm.cmd run smoke:result-review
npm.cmd run smoke:critical-values
npm.cmd run smoke:e2e-workflow
npm.cmd run smoke:report-output
npm.cmd run smoke:report-publish
npm.cmd run smoke:system-rules
npm.cmd run smoke:qc-events
npm.cmd run smoke:reagent-expiry-alerts
npm.cmd run smoke:infectious-alerts
```

检查重点：

- `git status -sb` 应显示预期分支和预期文件变更。
- `git ls-files --others --exclude-standard` 不应出现临时文件、构建产物、运行时数据库或下载资产。
- `git diff -- data/` 不应输出数据库或运行时数据差异。
- `db:reset` 和 `db:check` 必须通过。
- 十个 smoke 必须全部通过，任何失败都应阻止进入 RC 候选。

## 4. Electron 人工验证清单

RC 候选前必须打开 Electron 桌面应用并人工检查：

- 首页能看到 `demo_operator` 提示。
- 样本接收页面按钮状态合理；待签收、退样、补采等动作状态符合当前 seed/runtime 数据。
- 敏感动作会弹出确认提示，提示中包含敏感操作、demo_operator、真实系统角色权限约束和 audit log 说明。
- 结果审核页面正常显示，审核通过、驳回、报告生成、报告导出、报告发布入口仍符合原有流程。
- 危急值中心页面正常显示，通知、临床确认和完成闭环状态顺序合理。
- 报告发布 / 报告导出页面正常，未新增真实打印后端动作。
- 质控看板、试剂近效期、传染病阳性预警页面正常。
- 系统设置页能看到“敏感操作覆盖清单”。
- 覆盖清单包含 summary cards，并列出 17 个敏感动作。
- 覆盖清单能看到 UI gating、action guard、audit log、demo_operator、后端状态和说明。
- 页面导航正常，无明显布局破损。
- 中文正常显示，无乱码。

## 5. 禁止文件检查

以下文件或目录不应进入 v0.5 RC 检查 PR：

- `package.json`
- `package-lock.json`
- `database/schema.sql`
- `database/seed.sql`
- `src/database/**`
- `data/**`
- `dist/**`
- `build-log.txt`
- `build-dir-log.txt`
- 其他临时日志、构建产物、下载文件、导出文件、运行时数据库文件

如上述文件出现变更，必须先确认是否属于本步骤允许范围。默认情况下应阻止进入 PR。

## 8. 已知限制

- 当前不是生产级认证系统。
- `demo_operator` 仅用于桌面 demo，不是生产权限角色。
- UI gating 只是前端提示与确认，不等于真实安全认证。
- action guard 是 main 层最小入口保护，不等于完整后端授权体系。
- 当前未实现真实登录、真实用户角色切换、生产级授权策略。
- `reportPrint` 当前无真实后端打印动作，只在敏感操作覆盖清单中作为预留项说明。
- `systemRuleCreate` 当前只是 UI 预留，不新增真实 `system_rules` 写入。
- Electron 人工验证可能受 seed/runtime 数据状态影响，有些按钮不可点不一定是 bug，应结合当前记录状态判断。
- 当前仍不用于真实患者生产环境，不连接真实医院 HIS / LIS / EMR，也不连接真实检验仪器。

## 9. RC 候选判断标准

v0.5 可以进入 RC 候选前，至少应满足：

- 自动验证全部通过。
- Electron 人工验证无阻塞问题。
- Files changed 范围符合预期。
- 无禁止文件进入 PR。
- main 工作区干净。
- `data/` 无 diff。
- `dist/` 不提交。
- 文档边界说明清楚，未把当前系统描述为生产级医疗 LIS。
- 用户人工确认当前状态可作为 v0.5 RC 候选。

## 10. 后续 release 前仍需做的事

- 如需发布 v0.5 RC，需要另开 release / RC 步骤。
- 重新 build Electron 安装包和免安装版。
- 计算安装包和免安装包的 SHA256。
- 上传 GitHub Release。
- 用户下载 release 资产后再次校验 hash 和运行状态。
- 发布前再次确认 `dist/`、运行时数据库、构建日志和下载资产不会进入 Git 提交。
